MentisDB 0.9.1 — The Full-Feature Release

MentisDB 0.9.1 is the biggest release in the project's history. Four major features land together: GPT-4o memory extraction, federated cross-chain search, an official Python client, and a dramatically improved setup experience. This post covers all of it — the features, the design decisions, the documentation overhaul, and the full benchmark results.

What Is MentisDB?

MentisDB is an append-only semantic memory server for AI agents. Every thought an agent has — a preference learned, a decision made, a lesson earned — is stored permanently with a cryptographic hash chain that makes it tamper-evident. Retrieval is hybrid: lexical BM25, dense vector similarity, and a typed thought-graph that understands relationships like Corrects, Supersedes, ContinuesFrom, and BranchesFrom.

MentisDB speaks MCP (Model Context Protocol) natively over HTTP — no adapters, no bridges, no cloud dependencies. Point your agent at it and it bootstraps itself.

1. LLM-Extracted Memories

The most-requested feature is here. Instead of manually constructing ThoughtInput records, feed MentisDB raw agent output — conversation logs, terminal transcripts, reasoning traces — and GPT-4o (or any OpenAI-compatible endpoint) returns structured memory records ready for review and append.

Why Review Before Append?

LLM output is untrusted. Hallucinated facts, phantom preferences, and confident nonsense are real risks. The extraction pipeline returns raw ThoughtInput records — not committed thoughts — so you review them, optionally verify against source material, sign them with an Ed25519 key, then append. The hash chain is never polluted with untrusted content.

Configuration

Three environment variables. That's all: 

export OPENAI_API_KEY="sk-..."           # required
export LLM_BASE_URL="https://api.openai.com/v1"  # defaults to OpenAI
export LLM_MODEL="gpt-4o"                # defaults to gpt-4o

Rust API

use mentisdb::{LlmExtractionConfig, extract_memories_from_text};

let config = LlmExtractionConfig::from_env()?;
let result = extract_memories_from_text(raw_text, &config, None).await?;

// Review
for thought in &result.thoughts {
    println!("{:?}", thought);
}

// Append after review
for thought in result.thoughts {
    mentisdb.append_thought(agent_id, thought).await?;
}
println!("Tokens: {:?}", result.usage);

Under the Hood

The feature uses openai-rust2 — the same HTTP client powering cloudllm — with:

  • Connection pooling across extractions
  • Automatic 3× retry on 429 Too Many Requests and 5xx server errors
  • 60-second timeout
  • Temperature 0.1 for deterministic extraction
  • ResponseFormat::JsonObject — GPT-4o returns a JSON object, not a markdown code block

The extraction prompt instructs the model to return structured records matching MentisDB's ThoughtType taxonomy. Each extracted record includes a suggested thought_type, importance and confidence scores, and optional tags and concepts.

2. Federated Cross-Chain Search

Specialist agents shouldn't be siloed. With federated cross-chain search, a project-specific chain can still query the shared company-memory chain it branched from.

How Branching Works

Every chain can branch from any thought in any other chain. The new chain gets a genesis thought with a BranchesFrom relation pointing back to the fork point: 

POST /v1/chains/branch
{
  "source_chain_key": "company-memory",
  "branch_thought_id": "uuid-of-strategy-checkpoint"
}
→ { "chain_key": "project-x", "genesis_thought_id": "uuid" }

How Federated Search Works

ancestor_chain_keys() walks BranchesFrom relations to build the full ancestry of a chain. Ranked search on a branch transparently includes ancestor chain results — annotated with chain_key so you know where each result came from.

This enables a fleet memory hierarchy:

  • Base chain — company-wide knowledge, general agent preferences, onboarding facts
  • Project chains — branch at a checkpoint, accumulate project-specific memories
  • Queries — always search local + ancestors; never miss the base context

No configuration needed. Create a branch, keep querying the same way. The search backend discovers ancestors automatically.

4. Wizard Brew-First Setup

The interactive setup wizard is the fastest path to a working MentisDB integration. This release dramatically improves it for Homebrew users on macOS.

The mcp-remote Problem

Claude Desktop needs mcp-remote to connect to MentisDB's HTTPS MCP endpoint. mcp-remote requires Node ≥ 20 because it uses the File global from the Fetch API — available only in Node 20+.

Homebrew installs Node correctly. But npm-installed mcp-remote uses #!/usr/bin/env node, which may resolve to an old Node version on your PATH. The old Node crashes at runtime with ReferenceError: File is not defined.

What the Wizard Does Now

  1. Try brew install mcp-remote first
  2. If Homebrew mcp-remote is found, inspect its shebang — Homebrew writes a real path like #!/opt/homebrew/bin/node
  3. If the shebang points to a correct Node version, use mcp-remote directly as the command — no wrapper needed
  4. If npm-installed mcp-remote is found, wrap it with an explicit node path

Claude Desktop Config

Homebrew mcp-remote (Node ≥ 20, direct execution): 

{
  "mcpServers": {
    "mentisdb": {
      "command": "/opt/homebrew/bin/mcp-remote",
      "args": ["https://my.mentisdb.com:9473"],
      "env": { "NODE_TLS_REJECT_UNAUTHORIZED": "0" }
    }
  }
}

npm-installed mcp-remote (Node version uncertain, wrapped): 

{
  "mcpServers": {
    "mentisdb": {
      "command": "/usr/local/bin/node",
      "args": ["/usr/local/bin/mcp-remote", "https://my.mentisdb.com:9473"],
      "env": { "NODE_TLS_REJECT_UNAUTHORIZED": "0" }
    }
  }
}

The wizard handles all of this automatically. Run mentisdbd setup claude-desktop and answer the prompts.
